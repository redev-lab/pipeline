# 설계 노트 — `data/ingest/zone_boundary.py` (의제처리구역 → positive 라벨 경계)

> 규칙1. 첫 **positive 라벨**(지정 재개발구역)의 경계 + 시점 t를 만드는 리더.
> 상위 설계: [labels.md](labels.md). 출력은 labels.py 의 `_positives_from_zones`가 먹는다.

## 1. 하는 일 (①)
의제처리구역 SHP(UQ181)와 결정고시 CSV를 조인해 **(구역 폴리곤, 결정고시일=t,
소스메타)** 를 뽑는다. 재개발 유형만 남기고, 좌표계를 5186으로 통일하고, 한 구역이
여러 행으로 쪼개진 것을 합친다.

## 2. 핵심 난제 3개 — 실값으로 확정 (구현 전 해결)

### 2-1. 지뢰2 — 조인키 (★실데이터로 확정함)
- 의제처리 SHP엔 고시일이 없고 `NTFC_SN`(고시일련번호)만, 고시일은 결정고시 CSV.
- 교집합 테스트(4개구):
  - **`NTFC_SN` ∩ 결정고시 `고시관리코드` = 126/126 (100%)** ✓
  - `NTFC_SN` ∩ `고시번호` = 0, `WTNNC_SN`(의제 AGZ) ∩ 무엇 = 0
- 둘 다 포맷 `시군구5 + NTC + 날짜8 + 일련번호4`(20자)로 동일. → **조인키 확정:
  `UQ181.NTFC_SN == 결정고시.고시관리코드`**, 고시일 = 결정고시 `고시일자`.
- 구현 시 **매칭률 리포트**(재개발 구역 중 몇 %가 고시일을 얻었나) + 미매칭 목록.

### 2-2. 지뢰1 — 좌표계 (이미 만든 닻 재사용)
- 의제처리 = **EPSG:2097**(Korean1985/Bessel), 지적도 = 5186. 읽자마자
  `geo.to_target_crs()`로 **5186 재투영** 후에만 다룬다. (新 코드 없음, 닻 재사용.)

### 2-3. 재개발 필터 (★분포로 확정함)
- UQ181엔 택지·도시개발·소규모정비 등이 다 섞임. `SCLAS_CL`(소분류=사업유형)로 거른다.
- 4개구 분포 확인: **`UQ1221`(주택재개발 65건) + `UQ1222`(도시정비형 재개발 8건)
  = 73개** 만 재개발 positive. UQ1211/1212(주거환경), UQ181x(자율·가로·소규모재건축),
  None(존치·공공주택)은 제외. 코드↔의미는 DGM_NM 실텍스트로 확증("주택재개발사업구역").
- 필터 코드값은 매직넘버가 아니라 **데이터 코드체계** → 코드 상단 상수 + 주석으로
  명시(향후 모아타운 라벨 등 다른 클래스 필요 시 여기 확장).

## 3. 소스 추상화 (§11 — 핵심 설계 결정)
출력은 **소스와 무관한 표준 스키마**로 통일한다. v1=UQ181, 상업 전환 시 이 리더
함수 하나만 갈아끼우고 labels는 그대로(규칙7 config-driven과 같은 철학).

**표준 ZoneTable 스키마 (어떤 소스든 이걸 내놔야 함):**
| 컬럼 | 의미 |
|---|---|
| `zone_id` | 구역 식별자 = NTFC_SN(고시관리코드). ★labels의 R3 공간CV 묶음키 |
| `geometry` | 구역 폴리곤 (EPSG:5186) |
| `notice_date` | 결정고시일 (date). labels가 *연도*만 떼어 t로 |
| `zone_name` | 구역명(DGM_NM). 지뢰3(신통·해제 이름매칭)·사람 검증용 |
| `biz_type` | 사업유형(UQ1221/UQ1222) |
| `sigungu` | 시군구코드 |
| `source` | 소스 메타 태그 (예: "UPIS-UQ181-202602") — 추적성 |

## 4. 함수 분해 (④)
`data/ingest/zone_boundary.py`
| 함수 | 역할 |
|---|---|
| `load_zones(shp_path, gosi_csv_path) -> (gdf, report)` | 메인. 표준 ZoneTable 반환. |
| `_load_redev_polygons(shp_path)` | SHP 읽기 → 5186 재투영 → 재개발 필터 → 구역 dissolve. |
| `_load_notice_dates(csv_path)` | 결정고시 CSV → {고시관리코드: 고시일자} (cp949, 중복 처리). |
| `_join_notice_date(...)` | NTFC_SN 조인 + 매칭률 집계. |

## 5. 검토했지만 버린 대안 (③)
| 대안 | 기각 |
|---|---|
| `고시번호`로 조인 | 교집합 0. 실값이 부정. NTFC_SN=고시관리코드가 100%. |
| 좌표계 무시하고 바로 필지와 조인 | 2097↔5186 어긋남 → 라벨 오염(지뢰1). |
| SCLAS_CL 안 거르고 전부 positive | 택지·소규모재건축까지 재개발로 오학습. |
| 리더가 labels 스키마를 직접 알게 | 소스 교체 시 labels까지 수정. 표준 ZoneTable로 격리. |
| 중복 행을 그대로 | 한 구역이 멀티폴리곤·중복 레코드 → zone_id로 dissolve(1구역 1행). |

## 6. 검증
실데이터로: ① 재개발 73구역 추출 확인 ② 고시일 매칭률(목표 ~100%) + 미매칭 목록
③ 좌표계 5186 확인 ④ zone_id·notice_date·zone_name 샘플 육안 검증.

## 7. 미해결 / 주의
- 한 NTFC_SN에 변경·정정 고시가 여러 건이면 결정고시 CSV에 같은 고시관리코드가
  중복일 수 있음 → **최초 결정고시일**을 t로(또는 규칙 명시). 구현 시 중복 분포 확인.
- 고시일이 안 붙는 구역(NTFC_SN 결측·미매칭)은 positive에서 제외 + 리포트.
- 폴리곤 ∩ 지적 필지(→PNU)는 **labels.py**의 일(여기선 구역 폴리곤까지만 책임).
