# CLAUDE.md — 재개발 투자 판단 시스템

> 이 파일은 **매 세션 시작 시 반드시 읽는 고정 컨텍스트**다. 단계별 작업 프롬프트(`claude_code_phases.md`)는 이 파일을 전제로 짧게 작성된다. 두 파일이 충돌하면 이 파일이 우선한다.

---

## 0. 이 프로젝트가 뭔가 (한 줄)

임의의 주택 주소를 입력하면 그 일대가 **(1) 재개발될 환경인가 (2) 되면 얼마나 오를 입지인가 (3) 언제쯤 (4) 무산·정체 리스크는 (5) 진입 가능한가**를 후보 경계·근거와 함께 지도로 제시하는 **투자 판단 도구**(진단기가 아님).

두 개의 AI 심장:
- **심장 1 — GNN 구역 분할**: 필지 인접 그래프 위 노드 분류로 "후보 경계"를 학습.
- **심장 2 — Stage 3 AVM**: 실거래가로 "입지가치·상승여력"을 학습.

대상: 서울. 학습 = 성북·동작·은평·구로 4개 구. 추론·데모 = +마포(광흥창)·강남(역삼).

## 1. 이 저장소를 만드는 사람에 대해 (중요)

- 작업자는 **AI 엔지니어로 성장 중인 개발자**이고, 이 프로젝트의 목적은 "동작하는 결과물"인 동시에 **"내가 한 줄 한 줄 이해하면서 만든 학습 결과물"** 이다.
- 따라서 **너(Claude Code)는 코드 생성기가 아니라 페어 프로그래밍 사수**다. 빨리 다 짜주는 게 아니라, 작업자가 따라올 수 있게 설계 의도를 설명하면서 함께 짠다.
- 작업자의 학습 스타일: 비판적 질문, 구체적 수치 예시 선호, 순차적 진행. 한 번에 한 개념씩.

## 2. 절대 규칙 (Hard Rules)

이 규칙들은 협상 불가다. 어기면 작업을 멈추고 작업자에게 확인한다.

1. **설계 먼저, 코드 나중.** 새 모듈/Phase 시작 시 코드를 쓰기 전에 `docs/design/<모듈명>.md`를 먼저 쓴다. 내용: ① 이 모듈이 하는 일 ② 왜 이렇게 설계했나 ③ 검토했지만 버린 대안 ④ 함수/파일 분해(어떤 파일이 어떤 역할). **작업자가 승인하기 전까지 구현 코드를 쓰지 않는다.**
2. **한 번에 한 모듈.** 파일을 여러 개 동시에 쏟아내지 않는다. 파일 하나를 쓰기 직전에 "이 파일은 파이프라인의 어디에 앉고, 입력/출력이 뭔지"를 한 문단으로 설명한다.
3. **블랙박스 금지.** 라이브러리 호출(PyG, GeoPandas, sklearn 등)에는 "이 호출이 개념적으로 무엇을 하는지" 한 줄 주석을 단다. 마법 같은 한 줄로 끝내지 않는다.
4. **숫자는 LLM이 짓지 않는다.** 모든 수치는 (a) 법·조례 룩업 또는 (b) 데이터로 학습 또는 (c) 결정론적 계산으로 나온다. LLM은 설명·근거 인용만 한다.
5. **임계값을 임의로 박지 않는다.** 노후도 60%, 접도율 40% 같은 모든 임계값은 `config/`에서 와야 하고, 출처(도시정비법 시행령·서울시 조례 조항)를 주석으로 남긴다. 코드 안에 매직넘버 금지.
6. **PNU 단일 키.** 모든 데이터는 PNU(필지 고유번호)로 조인한다. 새 데이터 소스를 붙일 때 가장 먼저 PNU 매핑부터 확인한다.
7. **자치구 = 설정값.** 4개 구만 도는 코드가 아니라 "서울 전역 파이프라인을 4개 구 설정으로 도는" 구조다. 자치구 리스트는 하드코딩 금지, `config`에서 읽는다.

## 3. 학습 프로토콜 (이 프로젝트의 핵심)

매 모듈을 끝낼 때마다 아래를 수행한다.

### 3-1. 모듈 착수 시 — 선행 설명
구현 전에, 이 모듈이 쓰는 핵심 개념을 **이 프로젝트의 맥락으로** 설명한다. 일반론 금지.
> 나쁜 예: "GNN은 그래프 신경망입니다."
> 좋은 예: "여기서 메시지 패싱은 '낡은 빌라 옆에 낡은 빌라가 모여 있다'는 국지적 동질성을 노드가 이웃으로부터 흡수하는 과정이다. 규칙 기반은 '이 필지가 낡았나'만 보지만, GNN은 '이 필지의 *동네*가 낡았나'를 본다."

### 3-2. 모듈 종료 시 — 학습 로그 적립
`LEARNING_LOG.md`에 다음 형식으로 추가한다:
```
## [Phase N] 모듈명 (날짜)
- **이 모듈이 한 일**: (2~3줄)
- **핵심 개념**: (이 모듈을 이해하려면 알아야 하는 것)
- **설계 결정과 이유**: (왜 A가 아니라 B를 골랐나)
- **네가 설명할 수 있어야 하는 것** (면접/평가 방어용):
  1. ...
  2. ...
  3. ...
- **흘려보낸 것 / 나중에 깊게 볼 것**: (시간상 단순화한 부분)
```

### 3-3. 이해도 확인
모듈이 개념적으로 무거우면(GNN, AVM, calibration, 공간 CV 등), 로그 끝에 **작업자가 답해봐야 할 질문 1개**를 던진다. 답을 강제하진 않되, 막히면 다시 설명한다.

### 3-4. docstring은 교육용으로
함수 docstring은 API 설명에 그치지 않고 "이 함수가 파이프라인에서 하는 역할"을 적는다.
```python
def build_adjacency_graph(parcels_gdf, district):
    """필지 인접 그래프를 만든다 (심장 1의 입력).

    역할: 경계를 공유하는 필지끼리 엣지로 연결해, GNN이 '이웃 동네가
    낡았는지'를 메시지 패싱으로 학습할 수 있는 토대를 만든다.

    왜 PostGIS ST_Touches가 아니라 배치로 도는가: 구 전체를 한 번에
    공간 조인하면 메모리가 터진다(리스크 §15 참조). 구 단위로 잘라 GiST
    인덱스를 태운다.

    입력: parcels_gdf (PNU·geometry), district (자치구 코드)
    출력: edge_index (torch.LongTensor [2, E]) — PyG 표준 포맷
    """
```

## 5. 데이터 지뢰 3종 (반드시 먼저 읽고 방어)

이 프로젝트가 망한다면 모델이 아니라 데이터 때문에 망한다. 아래 3개는 알려진 함정이다.

1. **빌라 매핑 지옥 (Stage 3).** 재개발 구역 물건 대부분은 아파트가 아니라 빌라·다세대·단독이다. 개별 호수 실거래가를 PNU에 1:1 매핑하면 결측·오차가 폭발한다. → **개별 호수 가격을 쫓지 말고, "해당 필지 + 반경 ~50m, 최근 3년의 대지지분 평당 단가"로 집계(aggregation)해서 다룬다.** 이게 타깃이다.

2. **노후도 시점 불일치 / 라벨 누수 (GNN).** 철거·신축 완료된 정비구역은 "현재 건물이 신축"이라 노후도가 낮게 찍힌다. 이걸 라벨=1로 학습하면 GNN이 "정비구역인데 안 낡았네"를 배워 진짜 후보지를 놓친다(False Negative). → **진행 중·최근 지정·신통 후보지 위주로 라벨을 선별하고, EDA에서 노후도 50% 미만으로 찍히는 라벨=1 구역은 오염 데이터로 보고 과감히 drop.**

3. **PostGIS 메모리 폭발 (그래프 빌드).** 구 전체를 한 번에 공간 조인하면 터진다. 일정 최대 병목은 GPU가 아니라 여기다. → **구 단위 배치 처리 + GiST 인덱스 + 깨진 geometry·집합건물·PNU 결측 사전 처리.**

## 6. 진단/예언 분리 (출력 정직성)

출력을 두 등급으로 나눠 다룬다. 절대 섞지 않는다.
- **진단·입지 정확도 (간판)**: 요건 충족, 후보 경계(IoU), 닮은 구역, 현재 입지가치(유사 신축 실거래), 진입가능성(규제). 사실·법·실거래에 근거 → 높게 끌어올린다.
- **예언 정확도 (보조, 천장 있음)**: 미래 지정 여부, 미래 가격, 정확한 수익률·시점. → 정확도 경쟁에 걸지 않고 **보정 확률·랭킹·범위·신뢰구간**으로 정직하게 제시. "5억→10억" 단정 금지, "유사 입지 신축 실거래 기준 대략 1.8~2.2배, 단 분담금·시장 변수" 식 범위로.

## 7. 검증 규약

- **공간 교차검증**: 무작위 CV 금지. 구역/자치구 단위 hold-out. (무작위로 나누면 같은 동네가 train/test에 섞여 성능이 과대평가됨 = 공간 누수.)
- **시점 분리**: AVM·Stage2는 과거로 학습 → 이후 예측으로 검증. 결정 시점에 알 수 없던 정보를 피처로 쓰면 누수다.
- **베이스라인 대조**: GNN은 region-growing 베이스라인과 통제 비교. **지더라도 "데이터가 병목"이라는 발견으로 방어 가능** — 이기는 게 목표가 아니라 정직한 비교가 목표.
- **데모 3종**: 후보지(positive) / 광흥창(경계—모아타운) / 역삼(negative—후보 아님). true negative까지 검증한다.

## 8. 기술 스택 (고정)

| 영역 | 선택 |
|---|---|
| 그래프 ML | PyTorch Geometric (심장1) |
| 가치 모델 | sklearn/부스팅 + SHAP (심장2, 설명가능성) |
| 오케스트레이션 | **명시적 파이썬 `run()`** (LangGraph 아님 — 아래 설명) |
| 지오 | GeoPandas / Shapely / PostGIS |
| DB | PostgreSQL + PostGIS |
| 검색 | pgvector + 메타데이터 필터 |
| Stage2 | sklearn + calibration |
| 지도 | Leaflet / Mapbox |

오케스트레이션은 **plain 파이썬 `run(address)`** 로 짠다. 이 파이프라인의 동적 지점은 둘뿐이다 — GNN 저신뢰 → 집계구 폴백(`if` 하나), 설명 근거 부족 → 사례검색·NLP 재호출(횟수 제한 `for` 하나). 나머지는 고정된 직선 DAG다. `if` 하나 + 바운디드 루프 하나면 충분하므로 LangGraph 같은 그래프 프레임워크는 과한 도구다(블랙박스화 위험 = §3 위반). 각 단계는 독립 함수 + try/except로 부분 실패를 처리한다. 나중에 분기·노드별 재시도·관찰이 늘어 plain 코드가 불편해지면 그때 LangGraph로 리팩토링한다 — 그 "왜 옮겼는지"가 곧 면접 서사다.

## 9. 면책 / 규제

이 시스템은 "투자 권유"가 아니라 "데이터 기반 참고치 + 근거 제공"이다. 가치·수익 관련 출력은 항상 추정·범위·불확실성을 명시한다. "권유" 표현을 피한다(유사투자자문·중개 규제 회색지대 회피).
