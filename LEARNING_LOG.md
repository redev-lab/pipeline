## [Phase 0] 골격 + config 계층 + PNU 표준화 (2026-06-10)

- **이 모듈이 한 일**:
  §4 디렉토리 골격(빈 패키지)을 세우고, 파이프라인의 바닥 두 장을 깔았다.
  (1) config 계층 — 자치구·법정 임계값을 코드 밖 YAML로 빼고 단일 입구
  로더(`redev.config`)로만 읽게 했다. (2) PNU 표준화 계층(`redev.data.pnu`)
  — 출처가 제각각인 필지 식별자를 19자리 표준 문자열 하나로 모으는 함수들.

- **핵심 개념**:
  - **PNU(필지 고유번호) = 19자리 단일 조인 키.** 구조: 법정동코드10(시도2+
    시군구3+읍면동3+리2) + 필지구분1(1=일반,2=산) + 본번4 + 부번4.
    시군구코드(앞5)가 자치구 키.
  - **float64 안전 정수 한계 = 2^53 ≈ 9.0e15(약 16자리).** 19자리 PNU(~1.1e18)는
    이를 넘으므로 float로 표현하는 순간 끝자리가 반올림돼 *복구 불가*로 깨진다.
  - **단일 입구(narrow gate) 패턴.** 설정을 읽는 코드를 한 곳으로 좁혀,
    "단일 출처"를 규율이 아니라 구조로 강제한다.

- **설계 결정과 이유**:
  1. **PNU는 문자열, float은 거부(coerce 아님).** 처음엔 "무엇이 들어와도
     문자열로 고정"이라 썼는데, 작업자가 "float을 조용히 zero-pad하면 이미
     깨진 값을 *그럴듯한 가짜 PNU*로 만든다"고 지적 → 방어를 2층으로 분리.
     1차(진짜 방어선) = ingest에서 `dtype=str`. 2차 = `normalize_pnu`가
     float을 받으면 raise. **normalize는 손상을 복구하는 함수가 아니라
     손상이 일어났음을 폭로하는 함수다.**
  2. **자치구·임계값은 코드가 아니라 config.** Enum 하드코딩 대신 YAML +
     로더. "서울 전역 확장 = districts.yaml 한 줄 추가"가 성립하려면 읽는
     입구가 하나여야 한다.
  3. **임계값은 값과 출처를 같은 자리에.** YAML 주석에 조항 출처 + "현행
     재확인 TODO"를 값 바로 옆에 붙여, 6개월 뒤 "이 0.6 어디서 왔지"를 막는다.

- **네가 설명할 수 있어야 하는 것** (면접/평가 방어용):
  1. **왜 PNU가 단일 키인가?** 건물GIS·실거래가·정비구역경계는 각자 다른
     식별 체계를 쓴다. PNU(필지 단위)로 통일하지 않으면 "이 필지의 노후도 +
     이 필지의 실거래 + 이 필지가 정비구역인가"를 한 행으로 못 묶는다. GNN의
     노드도, AVM의 집계 단위도 결국 필지 → PNU가 없으면 두 심장의 입력이
     서로 다른 좌표계에 떠 있게 된다.
  2. **왜 float이 지뢰인가 (수치로):** PNU 1,129,010,100,100,100,000은
     2^53(9,007,199,254,740,992)보다 약 122배 크다. float64는 이 범위에서
     정수를 1단위로 표현 못 해 끝자리가 뭉개진다. CSV를 dtype 지정 없이 읽으면
     pandas가 float로 추론 → 조용히 깨진 키로 조인 → 틀린 필지에 데이터가 붙는
     최악(§5-3). 그래서 "읽는 시점"에 str로 막는 게 핵심.
  3. **config-driven이 어떻게 확장으로 이어지나:** 코드는 "training/inference
     자치구 코드 집합"이라는 *추상*에만 의존하고, 그 집합의 내용은
     districts.yaml이 채운다. 노원구를 추가하려면 코드 한 줄도 안 고치고
     YAML에 `{name: 노원구, sigungu_code: "11350"}` 한 줄만 넣으면 된다.
     "4개 구 파이프라인"이 아니라 "N개 구 설정으로 도는 서울 파이프라인".

- **흘려보낸 것 / 나중에 깊게 볼 것**:
  - `is_valid_pnu`는 형식(19자리·숫자·필지구분 1/2)만 본다. 실재하는 필지인지,
    법정동코드가 실제 존재하는 동인지는 검증 안 함 → 실데이터 조인이 결정.
  - 산(임야) 필지는 분해만 하고 필터링은 호출부에 위임. 실제로 산 필지를
    어디서 걸러야 하는지는 Phase 1 ingest에서 데이터를 보고 판단.
  - legal_thresholds.yaml의 값/조항은 전부 "재확인 TODO" 상태. 접도율의
    정확한 정의(기준 도로폭), 과소필지 면적 기준 등은 현행 조례 원문 확인 필요.
  - numpy.int64는 받지만(numbers.Integral), 그 int가 위에서 `int(float)`로
    만들어졌다면 2차 방어선도 못 잡는다 → 1차(dtype=str)가 반드시 있어야 완결.

- **이해도 확인 질문** (CLAUDE.md §3-3):
  > 누가 `transactions.csv`를 `pd.read_csv(path)`로 dtype 지정 없이 읽고,
  > PNU 컬럼을 `df['pnu'].astype(str)`로 바꾼 뒤 `normalize_pnu`에 넣었다.
  > `normalize_pnu`는 str을 받았으니 에러 없이 통과한다. 그런데도 이 PNU는
  > 깨졌을 수 있다. **왜인가? 그리고 이 사고를 어디서 막았어야 했나?**
  > (힌트: float→str 변환 순서를 따라가 보라.)

### ★ t-전쟁 — positive 라벨의 시점(t) 확보 (2026-06-11, Phase 1 최대 격전)
- **이 모듈이 한 일**: 의제처리 73개 재개발 구역에 "최초 정비구역 지정일(t)"을
  붙였다. 처음엔 보류(t 미확보) 66%(48/73) → 최종 **4%(3/73)**, 70개 t 확보.
- **핵심 개념 — 왜 t가 R1의 생사인가**: t는 "노후도를 어느 시점으로 계산할지"의
  닻이다(키스톤 §2-2). 잘못된 t(예: 최신 변경고시일)를 쓰면 2008 지정 구역이
  t=2025로 찍혀 R1 누수가 *정확히* 재발한다. 그래서 t는 절대 추정으로 안 채우고
  **소스를 늘려 회수**했다.
- **설계 결정과 이유 (소스 우선순위 = earliest-genuine)**:
  - OA-20283(결정고시) **이력 마이닝**이 1차 — NTFC_SN→토큰→동명가드→`MIN(결정)`.
    (조인은 됐지만 그게 주는 *최신 변경*을 거부하고, *전체 이력의 최초 결정*을 캤다.)
  - 촉진 sub는 부모 촉진지구 결정일(promotion_parent, 근사), keyless 단일지정은
    NTFC 직접(ntfc_direct: "지정"&¬"변경"), 공공재개발은 선정일(public_redev).
  - 불일치는 **전 소스 통틀어 earliest-genuine 채택 + t_alt 보존**(감사 가능).
- **네가 설명할 수 있어야 (면접 방어)**:

## [Phase 1] labels.py — (필지, 시점) 키스톤 완성 (2026-06-11)
- **이 모듈이 한 일**: Phase 1 최종 산출물. ZoneTable(positive 구역+t) ∩ parcels +
  신축파생·노후미지정(파생) + 해제(neg)를 합쳐 **99,964행 (필지,시점) 테이블**.
  positive 17,977 / reliable_neg 24,497 / uncertain 57,490. 수검 통과.
- **핵심 개념**: **WHERE/WHEN 분리** — labels는 t를 *재계산하지 않는다*. positive의 t는
  zone_boundary(OA 마이닝·earliest-genuine)가 끝냈고, labels는 ZoneTable을 받아 ∩parcels만.
  파생 neg의 t=현재. R2(오염 플래그)·R5(같은 PNU 다른 t 두 행)·충돌해소(같은 (pnu,t)
  우선순위)가 여기서.
- **설계 결정과 이유**:
  1. **노후도 배치는 벡터화**(aging.old_ratio_by_parcel: groupby.apply→sum/mean). 7.5만
     필지 R2/파생 분류를 apply로 돌면 분 단위 → 벡터화로 39초 전체. DRY 위해 aging에 둠.
  2. **겹치는 구역은 earliest t**: 한 필지가 두 구역에 걸리면 더 이른 t 채택(R1 보수).
  3. **R5는 코드 분기 없이 자연 발생**: 해제(neg,t1)+지정(pos,t2)이 다른 t라 충돌해소가
     안 합치고 두 행 남긴다. 실측 PNU `…233042`: 2007 지정 + 2018 해제 = 두 행.
- **네가 설명할 수 있어야**:
  1. "labels가 t를 안 만드는데 어떻게 시점 라벨인가?" — t는 상류(zone_boundary/해제/파생)가
     이미 박았고 labels는 *조립·∩·충돌해소*만. 관심사 분리.
  2. "positive:negative ≈ 18K:82K ≈ 1:4 — 이걸 어떻게 다루나?" — R8: focal/weighted loss
     + PR-AUC·F1, 정확도 금지. uncertain 57K는 R4 PU로 가중차등/제외.
